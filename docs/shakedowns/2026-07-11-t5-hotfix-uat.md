# Tabletop Shakedown: T5 — Incident/Hotfix + UAT Flows

**Date:** 2026-07-11 | **Mode:** live read-only inventory (Part 1) + synthetic incident walk (timeline) | **Run log:** `.ai-sdlc/runs/2026-07-11-tabletop-shakedown-t5-hotfix-uat.md`
**Hypothesis:** Explore the incident/hotfix and UAT work shapes with real KDP usage of the Hotfix*, UAT*, Clarification, and Risk issue types plus a synthetic mid-sprint production incident, to discover whether the pipeline has any incident entrance and whether these flows bypass the DoR/test skills entirely.
**Scenario source:** JQL inventories over KDP (hotfix family, UAT family, Clarification, Risk), SSQ System Malfunction sample, CQL sweep for hotfix pages (NDTW/KRN/DE/KS), fixVersion 1.69-OCT28 contents; synthetic spec: "advisor-facing document upload fails in production Tuesday of sprint week 2; root cause is a regression from a story shipped last sprint; fix ships as a hotfix mid-sprint."

## Headline

The hypothesis held, and then some: the pipeline has no incident entrance — and neither, anymore, does KDP's own issue-type registry. Only **5 hotfix-type issues exist in all of KDP history** (none since Nov 2024, one of them literally titled "Test Story"), while hotfix releases actually ship roughly monthly (8+ SRE deployment/release-notes pages, Oct 2025–Jun 2026) carried by **regular Story/Bug items with a fixVersion, on a 1–4-day clock**. The real detection channel is a different project entirely — SSQ "System Malfunction," ~10 new incidents per day — which no skill reads. The UAT types are similarly unprotected: across the entire UAT family, **exactly 2 issues have ever had a populated AC field**, and a representative UAT bug's full description is "Order ID: I33-CFLK-BFF." The library's protections are real but all of them are parked at an entrance these two flows never pass through.

## Part 1 — Content findings (defects in the scenario's Jira/Confluence)

Each finding names the skill whose rules surfaced it. Inventory basis: all 5 hotfix-family issues (complete), 50 most-recent UAT-family issues (sample; more exist), all 14 Clarifications (complete), all 27 Risks (complete), 10 recent SSQ System Malfunctions, 10 hotfix Confluence pages, both issues in hotfix release 1.69-OCT28.

### Blocking-grade

1. **The hotfix issue types are dead in practice but alive in the registry.** 5 hotfix-type issues ever; zero created since 2024-11-11 (KDP-35373). Meanwhile hotfix releases ship ~monthly: "HOTFIX Release 1.69-OCT28 Deployment" (NDTW), "1.72 Hotfix Release Notes" (KRN), "Release 1.80-MAR31-HOTFIX" (NDTW), "1.85 - JUN 05, 2026 Hotfix Release Notes" (DE)… The 1.69-OCT28 release contained a regular **Story** (KDP-38753) and a regular **Bug** (KDP-38742) — no hotfix type anywhere. Any skill that keys behavior off issue type — `bug-report-writer`'s type selection, `jira-confluence-writer`'s registry ("Hotfix Story/System Story: process types, pipeline reads, never creates") — is keyed to a vocabulary the team abandoned. *(surfaced by the tabletop's registry-vs-reality diff; owning references: `kdp-schema.md`, `kdp-bug-types.md`)*
2. **Work ships to production with no testable contract.** 0 of 5 hotfix-family issues have AC (`customfield_14705` null on all). KDP-38742, a production bug fixed and shipped in the 1.69-OCT28 hotfix in 4 days, has no AC and a two-sentence description; nothing states what "fixed" observably means. `test-plan-generator`'s contract ("The AC are your contract… you derive; you never invent expected behavior") is unmeetable on every hotfix item that ever existed. *(test-plan-generator; definition-of-ready-critic item 4)*
3. **UAT bugs are observations, not bug reports.** 36 of 36 sampled User Validation UAT Bugs have an empty AC field; none cite the AC they violate; descriptions are minimal to the point of vapor — KDP-35387's entire description is "Order ID: I33-CFLK-BFF" (no repro, no expected-vs-actual, no environment). Across the **entire** UAT family in KDP history, exactly 2 issues have AC (KDP-40341, KDP-16769 — and the latter's "AC" is a user-story sentence). Every rule in `bug-report-writer` ("minimal repro… expected comes from an AC… cited") is violated by essentially 100% of real UAT bugs. *(bug-report-writer)*
4. **UAT work is structurally orphaned.** Most sampled UV UAT Bugs have no parent; traceability to the feature under test exists only through ad-hoc "Blocks"/"Action item" links to the test case that broke (e.g., KDP-35387 → KDP-34794). Nothing links UAT bugs to the story/epic whose behavior they dispute, so nothing that walks a story's children (DoR critic on an epic, test-plan-generator, release-notes scoping) will ever see them. *(definition-of-ready-critic item 9; backlog-hygiene-auditor orphan check)*
5. **All 5 hotfix items are Done with the resolution field unset** (resolutiondate null despite Done/Closed status). Time-to-resolve for the one flow where speed *is* the point is unmeasurable from the data. *(sprint-report-generator — "every number in the report can be traced to a Jira query" fails here; backlog-hygiene-auditor)*

### Significant

6. **Hotfix parentage is a container, not a trace.** Hotfix items are parented to quarterly "Unplanned Work" epics (KDP-35036 "2024 Q4 Unplanned Work - AC Apps", KDP-31454 "2024 Q1…") — a bucket pattern that records *that* unplanned work happened but severs the link to the feature that regressed. *(impediment-radar's dependency lens; DoR critic item 9)*
7. **A test issue lives in production data**: KDP-32710, type Hotfix Story, summary "Test Story," no description, status Done. 20% of the hotfix corpus is noise. *(backlog-hygiene-auditor)*
8. **The Risk type is a single fossilized register.** All 27 Risks belong to one epic (KDP-25517 "NGP Risks and Issues"), bulk-created March 2023 by one PM; 16+ sit in non-terminal statuses ("Assessing," "In Mitigation," "Blocked") untouched for ~3 years. Risk-shaped signals since then (e.g., the .NET 10 epic's cross-team dependency, per the prior shakedown) never became Risks. *(impediment-radar — its "dependency rot" check would fire on the register itself)*
9. **Clarification is dormant.** 14 ever; 12 from 2020–2022, one since (2025). Several sat unresolved for years (KDP-561: created 2020, closed "Not Required" 2023). Team questions now live in comments and chat, invisible to any query. *(refinement-facilitator / DoR critic item 8 have no feed)*
10. **UAT test cases are one-shot campaign artifacts.** The 11 UV Test Cases were bulk-created over two days (2024-09-13/16) under a dedicated E2E epic (KDP-34699), run once to status "Pass," and never touched again; the sole 2026 PV Test Case (KDP-40541) is an orphan still Open. There is no maintained regression suite in Jira — each UAT is rebuilt from scratch. *(test-plan-generator's persist-as-Test-Case mode inherits this pattern unless told otherwise)*

### What the content gets right

- **The UAT campaign has real, invented-by-humans structure**: test cases parented to an E2E epic, bugs linked back to the exact test case that broke, business testers (advisors' staff, ops) filing same-day, fix turnaround of days-to-weeks. The traceability instinct is present — it just lives in link types instead of parents and AC.
- **The instance's link vocabulary is better than the library's.** KDP-32165 traces "Root Cause Fix" → SSQ-2793/SSQ-2812 (the System Malfunction incidents it fixes); KDP-31813 links "Defect created" → KDP-31870, a Story spun up purely as "QA Planning and Execution in the lower environment" for the hotfix — the team hand-built hotfix QA tracking the library doesn't offer. Neither link type appears anywhere in `kdp-schema.md`.
- **Hotfix descriptions state root cause plainly** (KDP-32165: the 10-second ADV2B timeout; KDP-31813: the inadvertently-shipped model change causing duplicate contacts). The narrative rigor is there; the structured fields are empty.
- **The real incident intake works.** SSQ System Malfunction absorbs ~10 incidents/day including automated Azure Sev-1 alerts (SSQ-35104) and support cases with case numbers — fast, active, triaged through "Impact Mitigation" states. The entrance exists; it's just in another project and invisible to this library.
- KDP-38753 (the story in hotfix 1.69-OCT28) carries a genuinely measurable AC ("full re-index takes less than 10 minutes") — proof the express clock doesn't preclude a contract.

## Not exercised

- **Developer skills against a real repo** — `implementation-planner`, `copilot-handoff-packager`, `code-review-critic`, `pr-hygiene` were walked against their SKILL.md text and the synthetic timeline only; no repository or PR was available. Their timeline verdicts (steps 6–8) are paper verdicts.
- **`ac-playwright-scaffolder`** — needs the test repo; also no house-style AC existed anywhere in the hotfix/UAT corpus to scaffold from, which is itself the finding.
- **Live sprint/board data** — standup-digest, impediment-radar, and sprint-report were walked against their rules plus issue-level data, not a live sprint board; the "off-board invisibility" claim derives from the historical parenting pattern of hotfix items, not from watching a sprint.
- **SSQ project depth** — sampled 10 recent System Malfunctions to establish the intake's existence and tempo; its workflow, SLAs, and links back to KDP were not exhaustively mapped.
- **UAT family beyond the 50 most recent** — older campaigns (e.g., the 2022 NGP PV UAT wave that produced KDP-16769) were seen only where queries surfaced them; the AC-presence count (2) is complete, the linkage/orphan percentages are sample-based.

## Verdict

The hypothesis is confirmed in a sharper form than posed. It isn't just that hotfix and UAT *types* bypass the DoR/test skills — the types themselves were abandoned by the team (hotfix: dead since 2024; UAT: episodic campaigns), and the real flows now run through channels the library cannot see: SSQ System Malfunction for detection, regular Story/Bug + fixVersion for the fix, SRE Confluence pages for shipping. The pipeline's protections are concentrated in the middle of a flow (plan → PR → review) whose entrance and closure don't exist for incidents; the two skills that would add the most under pressure (DoR's contract, test-plan's derivation) are precisely the ones whose written entry rules guarantee they're skipped. T5-S1 (incident entrance) + T5-S3 (express contract) are the gap, and T5-S2 is a same-day fix to a reference that currently gives wrong advice. The next-highest-value scenario: **a replay of an actual recent hotfix release** (e.g., 1.80-MAR27→MAR31, two hotfixes in five days) once repo/PR access exists — it would grade the paper verdicts on steps 6–9 against what the developers actually did, and test whether two hotfixes in a week is a cadence problem the aftermath skills should have flagged.
